## 第一部分：定位

### 1.1 模組性質

- **專案下的「選配模組頁面」**：非每個專案都有。僅「大型開發案 + 進入生產/組裝廠階段」才掛載。
- 意味著 PM-Core 專案需要「**模組開關**」概念：某些專案啟用物料模組、某些不啟用。
- 與「專案模板分類（全新開發案 vs 設變案）」議題相關（架構文件導航架構節 N.6 已列為另案）。物料模組可能是「全新開發案模板」才有的分頁。

### 1.2 與現有功能的關係

| | 現有 PM-Core | 物料模組 |
|---|---|---|
| 管什麼 | 任務 / 排程 / 進度（WBS、甘特、看板） | 料件 / BOM / 庫存 / 缺口 |
| 資料模型 | task | 料件、機型、BOM 矩陣、庫存 |
| 維度 | 時間（工期、依賴、日曆） | 數量（需求、庫存、缺口、可組台數） |

兩者交集在「同一個 J 系列專案」，但資料模型不同，**不可硬塞進 task 結構**。物料模組是專案頁的獨立分頁，與看板/Gantt/清單平級或更深一層。

## 第二部分：v9.3 試算表計算邏輯拆解（已摸清）

### 2.1 計算鏈

```
唯一輸入：機台需求源（機型 × 階段台數）
   ×  適用性矩陣（哪個機型用哪些料，1/0）
        ↓ 相乘加總
   各物料需求量
        −  庫存（性試前庫存 − 性試耗用 = 性試後庫存）
        ↓
   整機缺口（缺口 = 待新組需求 − 性試後庫存，不足才補）
        +  品保樣 / 封樣 / 商檢樣（另購，不耗庫存）
        ↓
   採購小計
        ‖（平行）
   共用料競爭分配 → 實際可組幾台（外機可組分析）
        ‖（平行）
   EMI 重洗例外分支（獨立計算，不併入主表）
```

### 2.2 五張表角色

| 表 | 角色 |
|---|---|
| 機台需求源 | **唯一輸入區**：機型×階段台數 + 適用性矩陣（1/0） |
| 物料總表 | 彙總：庫存/性試耗用/缺口/品保/封樣/採購小計 |
| 外機可組分析 | **共用料競爭分配**：算「實際可組幾台」 |
| 電控盒EMI差異 | 重洗 vs 不重洗的差異量 |
| EMI例外 | EMI 未過的整機重洗，獨立計算不併主表 |

### 2.3 關鍵欄位公式（試算表已明列）

- 性試後庫存 = 總庫存(性試前) − 性試耗用
- 整機缺口 = 長期待新組需求 − 性試後庫存（負值歸零）
- 採購小計 = 整機缺口 + 品保樣 + 封樣
- 可組台數：獨立可組 = 各用料庫存 MIN；實際分配可組 = 考量共用料競爭後結果

## 第三部分：痛點（為何試算表扛不住）

1. **共用料競爭分配** — 最痛。N 個機型搶 M 個共用料，試算表只能手動試分配。
   - 實例：6.3kW 獨立可組 3 台，但 140 壓縮機（4 台搶 3 個）+ R 機隔板（5 台搶 4 個）是 6.3/7.3 共用料，實際只能組 2 台。
   - 程式可自動算最優分配（每型保底 N 台後按優先序分配）。
2. **不可預期的變動導致不斷重算**：
   - 物料短缺 → 改設計（BOM 變）
   - 臨時增加樣機（台數變）
   - 測試/品保樣品需求量變動
   - 每次一變要手動重拉公式、重算分配、檢查連動。
3. **特殊狀況分支多**（EMI 重洗、品保、封樣、商檢），試算表用分頁硬接，易漏。

## 第四部分：時間維度 — 採購+組裝 road map（核心需求，2026-06-08 升級）

> 原列為「未來機會」，使用者明確要求納為核心需求。這是本模組相對試算表的最大價值。

### 4.1 核心計算鏈

```
專案各階段組立需求（哪階段 / 組幾台 / 何時要組好）
   ↓ 撈該階段組裝時間點
組裝需求日
   − 物料 L/T（前置交期 Lead Time）
   ↓ 回推
該物料「最慢下單日」= 組裝需求日 − L/T
   ↓ 全料件彙總
完整採購 + 組裝 road map（時間軸）
```

**這跟排程引擎同源**：「組裝日 − L/T = 最慢下單日」的日期回推，等同排程引擎的 addWorkdays / 前置 lag / 工作日曆計算。物料 L/T 本質上就是一種「前置 lag」（等料時間）。

### 4.2 組立時間點來源（雙路徑，分層）

- **主路徑：關聯 WBS task**。組立需求關聯到該專案某階段的 task，撈 `task.plannedStart`。
  - 接上排程引擎 → task 改期，組裝日與下單日自動跟動。
  - 「系統撈關鍵字查預計開始時間」= 用關鍵字比對 task 名稱/階段找到對應 task。
- **逃生口：手動填日期**。關聯不到時手填組裝需求日。
- 此雙路徑 = 雙層值機制：撈到的是 computed，手填的是 override（同 §getEffectiveSchedule 哲學：手填優先、空白才撈）。

### 4.3 資料結構補充（時間維度）

- **料件主檔加 `leadTime` 欄**（L/T，採購交期，全域層）。
- **組立需求**：機型/階段 × 台數 + 「關聯 taskId（主）或手填日期（逃生口）」。
- **特殊需求升級**：安規樣機、長期運轉機等，從「數量」升級為「數量 + 需求時間點」，才能進 road map 回推。可額外新增欄位填需求量與時間。

### 4.4 產出

- 採購 road map：每料件最慢下單日、數量，依時間排序（哪些料快到下單死線）。
- 組裝 road map：各階段組裝時間點、該時點所需物料是否到位。
- 與專案排程連動：可考慮畫進甘特或獨立時間軸。

### 4.5 分期

- 第一版仍可先做數量維度（取代試算表的當下痛點），時間維度緊接其後。但資料結構從一開始就要預留 leadTime / taskId / 需求時間點欄位，避免之後重構。

## 第五部分：資料結構（2026-06-08 定案方向）

### 5.1 分層：設計 A — 料件/庫存全域層

**料件主檔與庫存放全域層（專案之上），跨專案共享一份。** 各專案引用全域料件、扣全域庫存。

理由：使用者確認跨專案搶同一批庫存是真實場景（前公司充電樁槍櫃案經歷）。庫存掛專案下會打架（庫存算誰的、扣誰的），全域層才乾淨，符合「單一真實來源」最高原則。

**全域 vs 顯示分層**（關鍵釐清）：
- 「全域」是資料層的事，不是顯示層。料件「140 壓縮機 庫存 8 個」是客觀事實，跟從哪個專案看無關。
- 顯示層仍分專案：專案頁傳入 pid，算「本專案對全域料件的需求/佔用視角」。同 B-1 甘特 scope 模式（同一份資料、傳 pid 篩視角）。
- 維護放全域層（一處改庫存全專案同步），需求/BOM 放專案層。

### 5.2 顯示層級

```
全域層（專案之上）
└─ 物料總覽頁：料件主檔、總庫存、跨專案搶料總表
專案層（專案之下，選配模組）
└─ 專案物料頁：本案機型/台數/BOM/需求
   + 對共用庫存的佔用視角（可用N、本案需X、他案佔Y、淨給本案Z）
   + 可組台數 + 採購/組裝 road map
```

### 5.3 資料實體（暫定）

- **料件 Material**（全域）：料號、名稱、廠商、**庫存量**、單位、**leadTime（L/T）**、適用機型。
- **機型 Model**（專案）：型號、配置。
- **BOM 矩陣**（專案）：機型 × 料件 → **用量**（可手動設定，預設 1；非 1/0，支援一台用多個）。
- **需求源**（專案）：機型 × 階段 → 台數。
- **組立需求**（專案）：機型/階段 × 台數 + 關聯 taskId（主）/ 手填日期（逃生口）。
- **特殊需求**（專案）：品保樣/封樣/商檢/EMI重洗/安規樣機/長期運轉機 — 數量 + 需求時間點（另購、不耗庫存或另計）。
- **庫存異動**（未來）：到料、耗用、退回。

### 5.4 雙層值機制（核心骨幹，套用每個數字）

繼承排程引擎 getEffectiveSchedule（override > computed、帶來源）的設計，套到物料每個數字：

- `computed`：公式算出，唯讀。
- `override`：人手改，可選，**帶 reason + 時間**（解決 v9.3「改完不知為何改」的痛）。
- `effective = override ?? computed`：顯示用。
- UI 每格兩動作：①覆寫此格（手改，記原因）②還原公式（清 override，回吃 computed）。
- 被手改的格 UI 有視覺標記 + hover 顯示原因。
- **發現結果不對的兩條路**：①還原公式 ②回源頭改輸入讓全鏈重算（治本，呼應「源頭改對勝於中間補丁」）。
- 價值：手改是「蓋一層」，底下 computed 永遠在，隨時看「原本該是多少」、一鍵還原、可追溯。

### 5.5 分配演算法（可調規則）

共用料不夠分時的分配規則**可切換**（使用者要求）：需求多優先 / 交期急優先 / 手動指定…。

設計同 applySchedule(tasks, scope)：策略當參數傳入，演算法可插拔。`computeBuildable(materials, demands, strategy)`。

### 5.6 分層紀律

核心純函式只回傳資料、不碰 DOM/Storage：
- `computeMaterialDemand`（需求源 × BOM → 需求）
- `computeShortage`（需求 − 全域庫存 → 缺口/採購量）
- `computeBuildable`（共用料競爭分配 → 可組台數，strategy 可調）
- `computePurchaseRoadmap`（組裝日 − L/T → 最慢下單日，接排程引擎）

UI 層 render 吃算好的資料。
