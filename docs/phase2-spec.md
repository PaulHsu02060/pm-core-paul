## 3. 引擎1 排程引擎
- 從後端 `02_formulas.gs` 的 Sheet `WORKDAY()` 改為前端 JS 重寫: FS / SS / FF / SF + lag + WORKDAY（跳週末 + 假日 + 補班,讀工作日曆）,前置即時重算。
- 公式透明層: 計算搬前端後 hover 顯示真實算式。
- 相依衝突處理（今日定案）: 第一版做 **甲 + 乙**。
  - **甲（地基）**: `parsePredecessors` 解析前置 + `isTaskBlocked` 衝突偵測 + 警示（前置未完成 / 已提早,日期可能要調整,紅字提示,不自動改日期）。
  - **乙**: 自動順延「直接後續一層」,不連鎖。對短鏈 A→B→C: A 變 → B 自動移,C 偵測到衝突跳警示。
  - **丙（完整連鎖重排）**: 架構預留,後續啟用。共用地基 `parsePredecessors` / `isTaskBlocked`。

### 3.1 規則總綱（實作補充,對齊 `computeSchedule` / `docs/test-schedule-cases.js`）
- **排程模式 = 中間版**:
  - 手填的 `start`/`end` 引擎**尊重、不覆蓋**;只用 `isTaskBlocked` 偵測前置衝突跳警示。
  - 沒填 `start` 但**有前置** → 從前置 `end` 推算「建議 start」(另存,不蓋手填)。
  - **無前置又無 start** → 跳過,標「待排」。
- **工期天數（修正：直接讀 Sheet，不換算）**: `task.durationDays` = Sheet「工期」欄（G 欄），**本來就是工作天數**,不是小時。**不做** `ceil(estHours / dailyHours)` 換算。
- **end 算法**: `end = D.addWorkdays(start, durationDays - 1)`。⚠ `-1`:起算日算第 1 天,對齊 `test-workday-cases.js` 第 4 組(排程語意)。
- **estHours 保留不刪（待決點）**: 工時負荷引擎（引擎2）才需要「小時」。小時來源（補一欄真實工時 / 或 `工期天 × dailyHours` 反推）留到做引擎2 時再定,**今天不處理**;import 暫沿用舊值 `工期 × 6`。
- **里程碑一視同仁**: 類型欄「任務／里程碑」中,里程碑＝「階段文件審核流程」,**也有實際工期**。排程計算與一般任務相同（用 `durationDays`）,**不做 0 工期特例**。顯示層之後可標記（甘特菱形）,但排程邏輯不分。
- **predecessor 解析(`parsePredecessors`)**: 兩種格式都吃。
  - 純編號 `'5'` 或 `'5,6'` → `{dep, type:'FS', lag:0}`。
  - 含關係 `'5FS+2'` → 完整解析 `type`(FS/SS/FF/SF) + `lag`。
  - 空字串 / null → `[]`;未知格式不報錯,當 `FS` lag 0。
- **資料模型擴充(已於步驟1加欄)**: `task.predecessor`(前置編碼字串)、`task.wbs`(WBS 識別)、`task.parentWbsId`(子綁父)、`task.durationDays`(工期天數,工期單位修正後新增)。

### 3.1.1 computeSchedule 行為（最終定案：解法三 一鍵觸發連鎖）
- `computeSchedule(tasks)` 是**純計算函式**:產出「建議 start/end + 警示 + 連鎖建議清單」,**不直接寫回 task**。
- **手填日期一律尊重、不覆蓋**。
- 前置變動 → 重算後若發現「後續任務的建議值與現況不符」→ 標 **「待重排」** + 列出受影響的後續任務鏈。
- **套用連鎖由 UI「一鍵套用」觸發**(顯示層,本階段不做);運算層只負責算出「如果套用,後續該是什麼日期」。
- **乙 / 丙 的關係**:解法三的連鎖建議清單可以是「直接後續一層」(乙)或「整條鏈」(丙)。`computeSchedule` **算整條鏈的建議**,由 UI 決定一鍵套用的範圍。

### 3.2 已知狀態：新 Sheet 9 欄 vs 後端讀 14 欄（非 bug）
- 後端 `apps-script.gs` 寫死讀 **A:N 共 14 欄**;新複本 Sheet 目前只有 **9 欄(到 I「預計開始」)**。
- 欄位對映(前 9 欄完全對齊,無錯位):`A=N` / `B=PLM階段` / `C=子群組` / `D=任務名` / `E=類型(type)` / `F=前置(precedence)` / `G=工期(workdays→durationDays)` / `H=負責人` / `I=預計開始(plannedStart)`。
- **J~N(預計結束 / 實際開始 / 實際完成 / 進度 / 狀態)讀到空**,連帶:
  - `task.end` 空是**預期的** → 由排程引擎 `addWorkdays(start, durationDays-1)` 算出。
  - 實際開始 / 實際完成 / 狀態 → 由 **app 端回填**(對齊路線甲:Sheet 為單向真相來源,執行面資料在 app 累積)。
- **標注**:這不是 bug,是新 Sheet 設計如此。後 5 欄日後若要用,需在新 Sheet 補欄或另定資料來源。

## 4. 引擎2 工時負荷引擎（今日定案,本階段重點）
- 攤算: **模板A 均攤**。`estHours ÷ 跨越工作日 = 每日預設分攤`。模板B（每日上限）架構預留,第一版不啟用。
- 每日分段回填: 任務跨越的每一天一個下拉,0.5h ~ 8h（半小時一格）,填當天實際工時。
- 四個邊界規則:
  - **規則1(完成)**: 累計達總工時 = 完成,不需額外完成勾;按自動排程後,做完多出來的天「整個移除」,任務 bar 縮短（例 6h 原排 3 天,做了 4 + 2 = 6h,週三項目移除,bar 縮成 2 天）。
  - **規則2(溢出)**: 剩餘攤不下 → 最後一天顯示超量紅字預警,不自動順延日期。
  - **規則3(上限)**: 下拉 8h 是單格物理上限;爆量判斷看「各人跨專案當天加總 vs 每日容量」。
  - **規則4(重攤)**: 已填鎖定為事實,重攤只動未填的未來天;不允許累計超過該任務總工時。
- 負荷視圖: 各人每日跨專案工時加總,超過每日容量 → 爆量紅字。
- 註: 每日分段回填需在資料模型加一層「每日工時 log」（每 task 掛 N 天）,屬對鎖死欄位的擴充,需與月結邏輯對齊。
