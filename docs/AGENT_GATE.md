# AGENT_GATE — Claude Code 執行閘門（每個 session 開頭必讀，每步必守）

> 這份檔的存在理由：agent 有三個會害死專案的壞習慣 ——
> ① 做完一步自己往前衝、不等核可 ② 該貼原文只給摘要 ③ 報告測試/狀態時用「如上」「跑過了」帶過、不貼實際輸出。
> 以下規則是硬性閘門，不是建議。任何一條與「我想趕快做完」衝突時，閘門優先。

---

## 規則 0：閘門優先，禁止自我授權往前衝
- 你**只做我當前明確指示的那一件事**，做完**立刻停**，等我核可，才做下一件。
- 嚴禁「順手」多做：不在我沒要求時改別處、不預先做下一階段、不「我覺得反正等一下要做」就先做。
- 我說「停下」「等我核」「先別 commit」「先別碰下一層」時，**就是真的停**，不是讓你停一句話然後繼續跑。
- 你可以**建議**下一步是什麼（一句話），但**不准自己執行**它。

## 規則 1：一次一件，寫一處→貼 diff→等放行→才下一處
- 每改一個 hunk，貼該 hunk 的 diff，停。我說 OK 才改下一處。
- 不准把多處改動累積起來一次丟給我「整批 diff」（除非我明說要看整批）。

## 規則 2：報告一律貼原文，禁止摘要/「如上」/「跑過了」
- 我要測試結果 → 貼**該次指令的完整 stdout 原文**，不准寫「如上」「56 案過了」「結果一致」這種轉述。
- 我要 git status → 貼 git status 的**完整原始輸出**，不是你整理過的「只有 app.js」結論（結論可以加，但原文必須在）。
- 我說「貼 N 行原文不要摘要」→ 就貼 N 行原文，一字不改、不省略、不轉述。
- 「我上次其實有跑、只是沒貼」不是藉口 —— 沒貼原文＝沒做，重跑重貼。

## 規則 3：commit-gate 三步分開、逐一核可（最高優先，永不簡化）
1. **單獨**執行 `git status`，貼**完整原文**。我確認「只有預期檔、無 config.local.js / seed.local.js / seed.sample.js / _probe*.js 等機密/本地檔、無意外 untracked」後才往下。
2. 我放行 → `git add <明確檔名>`（不准 `git add .` 或 `git add -A`）。
3. 再**單獨** `git status` 看 staged 內容，貼原文，我確認。
4. 我放行 → `git commit`。
5. 我放行 → `git push`。
- **嚴禁**把 `git add` 與 `git commit`（或任何兩步）用 `&&`、`;`、`|` 串成一條指令。
- 每一步都要我獨立說「放行」才做下一步。

## 規則 4：commit 前驗證不可跳
- 動到邏輯/引擎的改動，commit 前必須 `node --check app.js` + `node docs/test-schedule-cases.js`，**兩者輸出原文都要貼**。
- 我指定要看的驗證沒貼原文之前，不准進 commit 流程。即使你「確定」會過，也要跑、要貼。
- 未線上互動驗證的功能，commit 訊息標 `[unverified]`。

## 規則 5：commit 單位 = 一個邏輯單元，切乾淨才開下一階段
- 一個階段（例：M2-T1 的 helper+A-1+A-2+ensureTaskType）是**一個 commit**。
- 這個 commit 沒 push 完，**不准開始下一階段**的任何改動（連「先看看」都不行）。
- 改動風險等級不同的工作（純加欄位 vs 改判斷邏輯 vs 動引擎）**必須切在不同 commit**，不准混在一起。

## 規則 6：跨機鐵則
- session 開工：先確認 cwd 是 pm-core-paul、remote 指向 pm-core-paul、`git log` HEAD 與遠端一致；不一致**先 pull**，不准在舊基礎上改。
- 收工/換機前：未完成工作也要 commit + push（可標 WIP）。

## 規則 7：CSS 鐵則
- 顏色/圓角/z-index/陰影一律走 style.css 的 `:root` 變數，禁止規則裡寫死 hex/數字。
- 合理例外：rgba 透明衍生、膠囊 99px、圓點 50%。
- **Design Tokens（2026-07-02 起）**：新開發的 Modal/表單/卡片，其 Modal 寬度（S400/M600/L800/XL1140）、間距（8px 網格：24/16/8）、字級（20-24/16/14/12/10-11 五級）、Icon（24/20/16）、元件高度（40/32/48）**一律從 `docs/UI-CSS-設計規範.md` §6 清單級距挑選，禁自定義 px**。既有 code 不回頭改、該區重做時順手收斂。

## 規則 8：改中文檔用 Edit
- 改含中文的檔一律用 Edit 工具，禁止用 PowerShell 文字回寫（會把 UTF-8 中文變亂碼）。
- commit 訊息避開 `>` `/`（PowerShell parser 會壞），用多段 `-m`。

## 規則 9：?v= 版本號只升動到的檔
- 只升「內容真的變動」的檔對應行，不要全部一起升（避免謊報快取）。

## 規則 10：底層需求（最高設計原則）
- 系統邏輯必須：**乾淨 / 正確 / 單一真實來源 / 不重複**。
- 「抽共用函式 vs 複製兩份」**一律選共用**。
- **移除功能＝順手清孤兒**：移除元素/功能時，因此變孤兒的 CSS/函式/import/變數**同 commit 一起清，不需逐次問**（移除前先 grep 確認只剩定義處、無其他引用＝真孤兒；連帶 `?v=` 版號該升就升）。屬機械批次、做完貼總結即可。
- 鐵則「不重構除非底層假設錯」防的是「對穩定不再動的 code 亂改結構」；
  但新需求若不打底會製造兩份重複時，**先合併再加＝正解**，不違反鐵則。

## 規則 11：架構文件三檔分工（2026-07-01 起）
- 主檔 `pm-core-architecture.md` **只放現役 spec**；已完成功能的落地紀錄／施工歷史／退役草稿一律進 `docs/archive/<功能群>.md`（`architecture-archive.md` 為 router）；全文地圖與「已完成功能去哪找」的反查看 `architecture-INDEX.md`。
- 功能落地後：落地紀錄搬對應 archive 群檔、主檔留現役 spec＋一行指標、**INDEX 同步更新該 § 的狀態與所在檔**。過時／退役內容（拆檔前 monolith 舊行號等）直接刪、不留。
- 動這三檔的**大段搬移**：用 node 腳本（錨點抓段＋dry-run 印計數證明＋utf8 讀寫＋split/join 保留換行），不用 Edit 硬框大段（坑5）；搬完 `git diff --ignore-all-space --shortstat` 與一般 shortstat 對齊＝無 CRLF flip（坑4）。找 code 以函式名為錨、檔案歸屬查主檔 §18.7.2。

## 規則 12：Mockup 定版即落細節進文件（防重工）
- 大功能（尤其 CSS/UI）**先出 Mockup 定版再寫 code**（HANDOFF §A 工作流程）。
- **Mockup 一定版，立即把細節完整回寫架構文件**：版面結構、欄位清單、逐條文案、色彩/`:root` token 對應、互動行為與狀態切換——讓**文件本身足以照著施工**，不靠翻對話。
- **禁止文件只留「大概方向」**：方向性描述不算定版 spec。過往痛點＝細節散在對話、每次重開 session 又重新討論 Mockup、一直重工。
- 落檔位置依規則 11 三檔分工（現役 spec 進主檔、施工細節/歷史進 archive、INDEX 更新）。

## 規則 13：共用是預設決策，別把「共用 vs 另寫」當問題來問（2026-07-02 定）
- 碰到「複用既有 render/元件/handler」vs「另寫平行版」時，**預設就是共用＋參數化差異**，不要停下來問 Paul A/B。Paul 明講：「能共用就共用、部分差異用改的，這是設計準則規範的，不需要來問我。」
- 實例：ECN 戰情室大表沿用 `_s2ListHtml`／`.s2-tbl`／`_s2PredCells`／既有 `_s2*` handler（hijack `_tplPreview` 指向 ECN res），只改欄位差異＋CSS scope 琥珀化；**不另寫 `_ecnListHtml` 讓 Table 肥大**。
- 呼應規則 10（抽共用 vs 複製一律選共用）＋memory `reuse-shared-no-hardcode`。技術上有侵入風險時（如動到共用 handler），用「加旗標分流、預設路徑零變更」保護既有功能，而非複製一份。

## 規則 14：UI 元件一律自適應，禁 overflow 爆版（2026-07-02 定）
- Paul 鐵則：「理論上表單都是自適應的，不該有 overflow 這問題。」新元件（HintBox/表格/卡片/彈窗）**寬度交給容器自適應**，長字走 `overflow-wrap:anywhere`＋`word-break`，**grid/flex item 一律給 `min-width:0`**（否則內容撐爆、不縮）。橫向捲動只掛在「該捲的那塊」（如大表包 `overflow-x:auto` 容器），別讓整個面板被寬內容撐開連累旁邊的說明框。詳見踩坑 坑10。

## 每步自檢（你回覆前先在心裡跑一遍，不通過就不准送出）
- [ ] 我做的是不是「只有我被指示的那一件事」？有沒有多做？
- [ ] 我該貼的原文（diff / stdout / git status）有沒有**完整貼出**，還是偷用了「如上」「跑過了」？
- [ ] 我是不是在某個「停下」之後又自己往前跑了？
- [ ] commit 相關步驟有沒有偷串指令、偷跳 git status、偷跳驗證？
- [ ] 這個改動該不該屬於現在這個 commit，還是混進了不同風險等級的東西？

任何一項打不了勾 → 停，重做那一步，貼原文，等核可。
